# IncidentIQ

An AI-powered Incident Intelligence Platform that automatically categorizes, prioritizes, and suggests resolutions for operational incidents using a local LLM (Ollama). Built as a polyglot, event-driven system with a React frontend, Spring Boot core service, FastAPI AI worker, and Kafka-based async processing.

## Screenshots

### Incident List — Dashboard View

![Incident List](docs/images/img_1.png)

The main incidents view shows all incidents in a sortable, filterable table with color-coded status badges (Open, Resolved) and priority indicators (P1-Critical, P2-High). Users can filter by status and priority, search with Cmd+K, and create new incidents. The table displays AI-assigned category (PAYMENTS, DATABASE, INFRA, etc.) and the original reporter. Seed data from resolved historical incidents appears alongside user-created ones.

### Incident Detail — AI Analysis & Resolution

![Incident Detail](docs/images/img.png)

The detail view shows the full incident with its AI-generated analysis. After an incident is created, the AI pipeline asynchronously processes it and populates:
- **Category** — automatically classified (e.g., PAYMENTS for a payment-related OOM issue)
- **Priority** — severity assessment with confidence score (P2-High, 80% confidence)
- **Resolution Suggestion** — actionable steps generated by the LLM using matched runbooks as context. The suggestion includes concrete commands, code snippets, and step-by-step remediation procedures pulled from the knowledge base. Markdown formatting (bold, numbered lists, code blocks) renders natively in the UI.

## What it does

1. **Users create incidents** via a React dashboard describing an operational issue (e.g., "Payment gateway returning 500s")
2. **Spring Boot** persists the incident, indexes it for search, and fires a Kafka event
3. **FastAPI** consumes the event and calls a local LLM (llama3.1:8b via Ollama) to:
   - **Categorize** the incident (PAYMENTS, AUTH, INFRA, DATABASE, NETWORK, UNKNOWN)
   - **Prioritize** it (P1–P4 with a confidence score)
   - **Suggest a resolution** by keyword-matching against runbooks and past resolved incidents, then prompting the LLM with the matched context
4. **Results flow back** via REST callback into Spring Boot, which updates the incident, re-indexes Elasticsearch, invalidates Redis cache, and publishes a completion event
5. **The React dashboard** polls and displays the AI analysis within seconds — no manual page refresh needed

| Layer | Technology | Purpose |
|-------|-----------|---------|
| Frontend | React 19, Vite, TypeScript, Tailwind CSS, React Query, Zustand | SPA with auth, incident CRUD, search, dashboard |

| Messaging | Apache Kafka (Confluent 7.6) | Event-driven async bridge between core and AI services |
| Infra | Docker Compose | Local development orchestration (9 containers) |

## Getting Started

### Prerequisites

- **Docker Desktop** (for infrastructure containers)
- **Java 21** (for Spring Boot)
- **Python 3.12+** (for FastAPI)
- **Node.js 22+** (for React frontend)

### Step 1: Start Infrastructure

```bash
cd infra
docker compose up -d \
  incidentiq-postgres incidentiq-redis incidentiq-elasticsearch \
  incidentiq-zookeeper incidentiq-kafka incidentiq-ollama
```

Wait for all containers to be healthy (~15 seconds). Verify:
```bash
docker ps --format "{{.Names}} {{.Status}}" | grep incidentiq
```

### Step 2: Pull the LLM Model

```bash
docker exec incidentiq-ollama ollama pull llama3.1:8b
```

This downloads ~4.7GB. AI features won't work until this completes. You can proceed with the next steps while it downloads.

### Step 3: Create Elasticsearch Indices

```bash
curl -s -X PUT http://localhost:9200/incidents -H "Content-Type: application/json" \
  -d @elasticsearch/mappings/incident-mapping.json

curl -s -X PUT http://localhost:9200/runbooks -H "Content-Type: application/json" \
  -d @elasticsearch/mappings/runbooks-mapping.json

curl -s -X PUT http://localhost:9200/service_metadata -H "Content-Type: application/json" \
  -d @elasticsearch/mappings/service-metadata-mapping.json
```

### Step 4: Start Spring Boot (Terminal 1)

```bash
cd backend-core
DB_USERNAME=incidentiq DB_PASSWORD=incidentiq \
JWT_ACCESS_SECRET=dev-access-secret-key-min-32-chars-long \
JWT_REFRESH_SECRET=dev-refresh-secret-key-min-32-chars-long \
AI_SERVICE_TOKEN=dev-ai-token \
./mvnw spring-boot:run -Dspring-boot.run.profiles=dev
```

Wait for `Started IncidentIqApplication` in the logs.

### Step 5: Seed Data

```bash
cd ai-service
source .venv/bin/activate   # or: python -m venv .venv && source .venv/bin/activate && pip install -r requirements.txt
pip install psycopg2-binary  # required for seed script
python scripts/seed_elasticsearch.py
```

Seeds 12 runbooks, 10 service metadata entries, and 25 historical resolved incidents.

### Step 6: Start FastAPI (Terminal 2)

```bash
cd ai-service
source .venv/bin/activate
AI_SERVICE_TOKEN=dev-ai-token uvicorn app.main:app --port 8000
```

### Step 7: Start React (Terminal 3)

```bash
cd frontend
npm install   # first time only
npm run dev
```

### Step 8: Use the App

Open **http://localhost:5173** in your browser.

1. Register a new account
2. Create an incident (e.g., "Database connection pool exhausted — HikariCP showing 0 available connections")
3. Watch the AI analyze it — the detail page polls every 3 seconds until classification completes
4. View the AI-generated category, priority, confidence score, and resolution suggestion
5. Add comments, view audit history, resolve the incident

## Key Features

- **JWT Authentication** with refresh token rotation and Redis-backed blacklist
- **Role-Based Access Control** (VIEWER, ENGINEER, ADMIN)
- **Incident State Machine** (OPEN → IN_PROGRESS → RESOLVED → CLOSED) with ADMIN override
- **Audit Trail** — every field change recorded with who/when/what
- **Redis Caching** — incident detail (10min), list (30s), dashboard (60s), rate limiting
- **Keyword Search** — Elasticsearch multi_match with title^2 boost
- **AI Classification Pipeline** — fully async via Kafka, never blocks the user request
- **Dead Letter Queue** — failed AI processing goes to DLQ, auto-retried every 5 minutes
- **Dark Mode** — system preference detection with manual toggle
- **Responsive Design** — usable down to 375px mobile width
- **Cmd+K Search** — command palette for quick incident search
- **Distributed Tracing** — X-Trace-Id propagated through HTTP → Kafka → FastAPI → callback

## Future Scope (MVP2)

These are intentionally deferred but the architecture supports them without rework:

| Feature | Description |
|---------|-------------|
| **Semantic Search** | Vector embeddings via `nomic-embed-text` + Elasticsearch kNN for similar-incident search |
| **Full RAG Pipeline** | Multi-hop retrieval + reranking before LLM prompting (possibly via LangChain) |
| **Live Runbook Sync** | Confluence/GitHub integration to keep runbooks updated instead of static seed files |
| **Document Ingestion** | Continuous sync pipeline for runbooks and knowledge base articles |
| **Real-time Updates** | WebSocket push for incident updates instead of polling |
| **Notification Service** | Consume `incident.ai.completed` events for email/Slack/PagerDuty alerts |
| **Analytics Dashboard** | Historical trends, MTTR tracking, category distribution charts |
